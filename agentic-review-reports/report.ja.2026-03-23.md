# Minishell 監査レポート

**[Result]**: **CRITICAL (FAIL)**

**[Forbidden Function Check]**: **PASS**
許可された外部関数以外の使用は認められず、仕様を遵守している。

**[Audit Items & Evidence]**:

### コンパイル
- **説明**: `-Wall -Wextra -Werror` フラグを用いてコンパイルし、不要な再リンクが発生しないこと。
- **エビデンス/テストコマンド**: `make re && make`
- **観察結果**: 正常。警告なしでビルドされ、再リンクも発生しない。

### 単純なコマンドとグローバル変数
- **説明**: 単純なコマンドの実行とグローバル変数の数（1つ以下）の確認。
- **エビデンス/テストコマンド**: `ls` の実行および `main.c` の確認。
- **観察結果**: 正常。グローバル変数は `g_sig` 1つのみである。

### exit
- **説明**: 引数あり/なし、および異常な引数での exit 挙動。
- **エビデンス/テストコマンド**: `exit "  42  "`, `exit 9223372036854775807`
- **観察結果**: **[FAIL]** 
    - `exit "  42  "` のように空白を含む数値を `ft_isnumeric` が拒絶し、エラーとなる（bashは許容する）。
    - `ft_atoi` を使用しているため、`long long` の最大値付近で数値が切り詰められ、正しい終了ステータス（255）が得られない。

### シグナル
- **説明**: プロンプト待機中および実行中のシグナル挙動。
- **エビデンス/テストコマンド**: Ctrl-C 入力。
- **観察結果**: **[仕様乖離]** 
    - `handler_interactive` 内で `write(STDOUT_FILENO, "^C\n", 3);` を実行しており、bashの標準的な挙動（単なる改行）と異なる表示が行われる。

### Heredoc (リソース管理)
- **説明**: Heredocで使用されるテンポラリファイルの管理。
- **エビデンス/テストコマンド**: `cat << EOF1 > /non_existent/fail << EOF2` 実行後の `/tmp/` 確認。
- **観察結果**: **[CRITICAL FAIL]** 
    - `free_ast` 関数がテンポラリファイルを `unlink` しないため、リダイレクト処理の途中でエラーが発生した場合や、シェルを `exit` した際に、未処理のテンポラリファイルがディスク上に永久に残存する（リソースリーク）。

### ワイルドカード
- **説明**: `*` によるファイル名展開。
- **エビデンス/テストコマンド**: `echo *`
- **観察結果**: **[仕様乖離/技術的不備]** 
    - `sort_str_array` が `ft_strcmp` によるASCII順ソートを行っており、大文字小文字の混在時にbashと異なる順序になる。
    - `append_match` において、マッチのたびに配列を再確保（`O(N^2)`）しており、かつ内部のエラー発生時に元の配列を `free` せずにポインタを捨てるため、メモリリークが発生する。

**[仕様との乖離]**:
1.  `exit` における空白入りの数値の扱いが bash と異なる。
2.  対話モードの Ctrl-C で `^C` という不要な文字列が表示される。
3.  ワイルドカードのソート順序が ASCII 準拠であり、シェルの標準（ロケール準拠）と異なる。
4.  多段パイプが再帰構造に依存しており、固定長の PID 配列を使用しているため、極端に長いパイプラインでの管理に懸念がある。

**[破壊的テストの結果]**:
- `cat << EOF1 > /err << EOF2` 等のコマンドで、エラー発生後に `/tmp/.minishell_heredoc_*` が残存することを確認。
- `exit 9223372036854775807` で期待される `255` ではなく、切り詰められた値が返ることを確認。

**[リファクタリング要件]**:
1.  **[必須]**: `free_ast` および `cleanup_and_exit` 内で、すべてのリダイレクトノードを走査し、`TK_HEREDOC` タイプのファイルが存在すれば `unlink` する処理を追加すること。
2.  **[必須]**: `exit` の数値判定を修正し、前後の空白を許容するとともに、`long long` をパースできる専用の `ft_atoll` 等を使用すること。
3.  **[推奨]**: `append_match` のアロケーション戦略を、2倍ずつ確保する方式に変更し、エラーハンドリング（メモリ解放）を徹底すること。

**[レビュイーへの説明要求事項]**:
1.  **テンポラリファイルのライフサイクル**: なぜ `free_ast` でのクリーンアップを実装しなかったのか。エラー時や終了時のファイル残存をどのように認識していたか説明せよ。
2.  **数値パースの型選択**: `exit` ステータスは `long long` 範囲の入力を受け付ける必要があるが、`int` を返す `ft_atoi` を使用した技術的判断の根拠を説明せよ。
3.  **ワイルドカードのソート基準**: bash と異なる ASCII 順でのソートを採用した意図はあるか、あるいはロケール（LC_COLLATE）の影響をどのように検討したか説明せよ。
4.  **リソース管理の堅牢性**: 子プロセス内（`exec_cmd`）で行われている不必要な `dup` 処理が、システムリソース（FD上限）に与える影響についてどのように評価しているか説明せよ。
