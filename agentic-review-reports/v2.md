# Minishell 監査報告書

- **[Result]**: FAIL
- **[Forbidden Function Check]**:
    - **禁止関数の疑い**: `rl_catch_signals` が使用されている。これは `readline` ライブラリのグローバル変数であるが、`en.subject.pdf` の許可関数/シンボルリストには一切記載がない。許可されていない外部シンボルの参照は仕様違反とみなす。

- **[Audit Items & Evidence]**:
    - **コンパイル**:
        - **内容**: プロジェクトが `-Wall -Wextra -Werror` フラグでコンパイルされ、Makefile に必要なルールが含まれているかを確認。
        - **証拠/テストコマンド**: `make -C project_dir`
        - **観測結果**: Makefile は存在し、正しいフラグが設定されている。再リンク（relink）も発生せず、この項目は合格である。
		
    - **ヒアドキュメントの管理**:
        - **内容**: ヒアドキュメント用に作成された一時ファイルが適切にクリーンアップされているかを確認。
        - **証拠/テストコマンド**: ヒアドキュメントを実行し終了した後、`ls /tmp/.minishell_heredoc_*` を実行。
        - **観測結果**: **FAIL（不合格）**。 `free_ast.c` 内で `unlink` が一度も呼び出されていない。ヒアドキュメントを実行するたびに、`/tmp` 内にゴミファイルが蓄積されるリソースリーク状態である。

    - **パス解決**:
        - **内容**: `PATH` 内の空のパス要素（カレントディレクトリを示す）の挙動を確認。
        - **証拠/テストコマンド**: `export PATH=:/usr/bin` を実行し、カレントディレクトリのコマンド実行を試行。
        - **観測結果**: **FAIL（不合格）**。 `ft_split_c` が先頭や連続する区切り文字をスキップするため、空のパス要素（本来 `./` と解釈されるべきもの）が無視されている。Bash との互換性を著しく損なう仕様逸脱である。

    - **メモリの安全性**:
        - **内容**: `malloc` 失敗時のエラーハンドリングとメモリリークの有無を確認。
        - **証拠/テストコマンド**: `append_char` の静的解析。
        - **観測結果**: **FAIL（不合格）**。 `append_char` は `malloc` が失敗した場合、元の文字列を `free` せずに `NULL` を返している。呼び出し側も `NULL` チェックを行っていないため、メモリリークとセグメンテーションフォールトの温床となっている。

- **[Deviations from Spec]**:
    - **リソースリーク**: ヒアドキュメントの一時ファイルが削除されない。
    - **Bash 互換性欠如**: 空のエントリを含む `PATH` のパースが不正確である。
    - **未定義動作 (UB)**: `exit.c` の `ft_isoverflow` において、オーバーフローを確認する前に符号付き整数のオーバーフローを発生させている。

- **[Destructive Test Results]**:
    - **多重ヒアドキュメント中の中断**: `ctrl-C` を入力すると、`/tmp` 内にゴミファイルが残留する。
    - **巨大な終了ステータス**: `exit 9223372036854775808` 実行時、`ft_isoverflow` 内で未定義動作が誘発される。

- **[Refactor Requirements]**:
    1. **一時ファイルの削除**: `free_ast.c` の `free_redirects` を修正し、一時ファイル名のパターンに一致する（またはヒアドキュメント型の）リダイレクトの場合、`unlink` を実行すること。
    2. **`ft_split_c` の修正**: 区切り文字が連続する場合でも空文字列（`""`）を保持するように修正し、`PATH=:...` が正しくカレントディレクトリを含むようにすること。
    3. **堅牢な `append_char`**: `malloc` が失敗した場合でも、入力文字列を適切に `free` するか、呼び出し側で完全なエラー処理を行うこと。あるいは、動的バッファを用いた実装へのリファクタリングを推奨する。
    4. **`ft_isoverflow` の修正**: 乗算を行う**前**に、現在の値が限界を超えないかチェックする論理（例：`res > LLONG_MAX / 10`）を導入し、未定義動作を完全に排除すること。

- **[Required Explanations from Reviewee]**:
    1. **rl_catch_signals**: この変数を使用した理由と、許可リストに含まれていないにもかかわらず使用が許容されると判断した根拠を説明せよ。
    2. **ヒアドキュメントのクリーンアップ**: なぜクリーンアップロジックに `unlink` の呼び出しが含まれていないのか。ゴミファイルの蓄積をどのように正当化するのか説明せよ。
    3. **整数のオーバーフロー**: なぜ `long long` の限界を超えないかを確認する前に、`res * 10 + digit` という計算を実行しているのか。未定義動作のリスクをどう考えているのか回答せよ。
